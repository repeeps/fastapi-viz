수식·코드 없는 직관 입문 · 0번 + 16챕터

FastAPI, 눈으로 보기

웹 서버는 요청 하나가 들어오면 안에서 보이지 않는 일을 잔뜩 해요 — 어느 핸들러로 보낼지(라우팅), 입력이 올바른지(검증), 필요한 걸 채워주고(의존성), 한 스레드로 수천 요청을 어떻게 동시에 처리하는지(이벤트 루프), DB 연결을 어떻게 빌려 쓰는지… 코드만 봐선 안 보이는 FastAPI 런타임의 안쪽을 움직이는 그림으로 익히는 0번 입문 + 16챕터.

👋 처음이세요?
0번 — 서버가 하는 일 (요청 하나의 여정)
브라우저가 요청을 보내면 서버 안에서 무슨 일이? 라우팅 → 검증 → 의존성 → 핸들러 → 응답. 전체 파이프라인을 한눈에. →

📚 처음이라면 0번부터 순서대로. 이건 문법 강좌가 아니라 멘탈 모델이에요 — 요청 카드가 파이프라인을 통과하고, 한 이벤트 루프가 여러 요청을 번갈아 처리하고, 의존성 트리가 채워지고, 커넥션 풀이 고갈되는 걸 직접 보며 "왜 이렇게 도는지"를 익혀요. 척추는 요청여정(A·B)→async(C)→의존성주입(D)→바깥세계(E)→운영(F). async 이벤트 루프(C)와 의존성 주입(D)이 FastAPI를 다른 프레임워크와 구분 짓는 핵심이에요.

PART A · 요청의 여정
01

HTTP 요청·응답

메서드·경로·헤더·바디로 된 "약속된 편지". 서버는 상태코드로 답한다.

methodstatus code
02

라우팅

이 URL은 어느 핸들러로? 경로·쿼리 파라미터, 매칭 순서(구체적 먼저).

pathquery
PART B · 입력과 출력
03

Pydantic 검증

들어온 JSON을 타입 있는 모델로. 틀리면 핸들러 도달 전 422로 막는 문지기.

pydantic422
04

응답 모델

내보낼 것만 거름망에 거른다 — 비밀번호 필드는 실수로도 안 나가게.

response_model
PART C · async와 이벤트 루프 (하이라이트)
05

동기 vs 비동기

왜 async? 한 직원이 굽는 동안 다음 손님을 받는다. 같은 3요청, 총 시간이 다르다.

asyncawait
06

이벤트 루프

한 스레드가 일을 번갈아. 동시성 ≠ 병렬성 — 실행은 늘 하나, 대기만 겹친다.

event loop
07

루프를 막지 마라

FastAPI 1순위 실수 — 동기 호출 하나가 전체를 멈춘다. 동기 작업은 def로.

blockingthreadpool
08

동시 요청과 백그라운드

한 루프가 수천 요청을 어떻게. 응답 후 도는 백그라운드 태스크의 정체.

concurrencybackground
PART D · 의존성 주입 (하이라이트)
09

Depends

핸들러가 "필요한 것"을 선언하면 프레임워크가 채워준다. 왜 좋은가.

Depends
10

중첩과 캐싱

의존성의 의존성(트리). 한 요청 안에서 같은 의존성은 딱 한 번만.

treecache
11

yield 생명주기

셋업 → 핸들러 → 정리. DB 세션을 열고 자동으로 닫는 보장된 흐름.

yieldcleanup
PART E · 바깥 세계
12

DB 세션·커넥션 풀

풀에서 연결을 빌리고 반납. 동시에 쿼리하는 요청이 풀보다 많으면 대기·고갈.

pool
13

N+1 쿼리

목록 1번 + 항목마다 1번씩 = 보이지 않는 쿼리 폭발. 일괄 조회 한 발로.

N+1
14

인증 (JWT)

로그인 → 토큰 → 매 요청 검증. 토큰은 암호화가 아니라 서명된 출입증.

JWTOAuth2
PART F · 실전·운영
15

미들웨어·에러 처리

요청이 들어가며·응답이 나오며 거치는 양파 껍질. 순서가 중요. 예외 → 깔끔한 응답.

middleware
16

배포와 워커

ASGI 서버, 워커(프로세스) vs async(동시성). 언제 워커를 늘리나 + 너머.

uvicornworkers
