PART F · 실전·운영 — CH 16

배포와 워커

코드를 짰으면 실제로 띄워야 해요. FastAPI 앱은 스스로 못 돌고, 앞단에 ASGI 서버(uvicorn 등)가 있어 요청을 받아 앱의 이벤트 루프에 태워줘요(0번의 문지기). 트래픽이 늘면? 두 가지 손잡이가 있어요. ① async(동시성) — 한 워커(프로세스) 안에서 이벤트 루프가 기다림의 틈을 메워 많은 요청을 받아내요(PART C). ② 워커(병렬성) — 같은 앱을 여러 프로세스로 복제해 띄우면(각자 독립된 이벤트 루프) 진짜 동시에 여러 요청을 실행해요. 핵심 구분 — I/O 대기가 많으면 async로 충분(워커 적게), CPU 계산이 많으면 워커를 늘려야 해요(보통 CPU 코어 수에 맞춰). async가 못 하는 "진짜 병렬 실행"을 워커가 채우는 거죠. 마지막으로 이 트랙이 안 다룬 것들과 다음 발걸음을 정리하며 마무리해요.

Step 00 — ASGI & Workers

문지기(ASGI) + 복제된 워커들

맨 앞 ASGI 서버가 들어온 요청을 워커들에 나눠줘요. 워커 수를 바꿔보세요 — 워커는 같은 앱의 복제 프로세스이고 각자 독립된 이벤트 루프를 가져요. 워커가 많을수록 진짜 동시에 처리할 수 있는 요청이 늘죠(각 워커는 또 자기 루프로 여러 요청을 동시 처리하고요).

→ 워커 = 프로세스(메모리 따로, CPU 병렬은 코어 수까지). async = 한 워커 안에서 I/O 대기를 겹쳐 받는 동시성. 서로 다른 차원이에요 — 워커는 계산을, async는 기다림을.

Step 01 — Workers vs Async

무엇을 늘려야 하나 — 일의 성격에 따라

일의 성격(기다림 위주 vs CPU 위주)을 바꿔보세요. I/O 위주(DB·API 대기): 한 워커의 async가 기다림을 메워 많은 요청을 받아내요(단 여러 코어를 다 쓰려면 I/O 앱도 워커를 코어 수만큼 — async는 "각 워커 안"의 동시성). CPU 위주(계산·이미지 처리): 한 루프로는 안 빨라지니(병렬 아님) 워커(프로세스)를 코어 수만큼 늘려야 진짜 동시에 계산해요(가속은 이상적일 때 ×워커수, 실제론 오버헤드로 그보다 덜). async는 기다림을, 워커는 계산을 — 서로 다른 손잡이예요.

→ 실무 출발점: I/O 위주 API는 워커 적게+async, CPU 위주는 워커를 코어 수에. 무거운 작업은 작업 큐로 빼고요(CH08).

Step 02 — Beyond

너머의 주제들 — 이 트랙이 안 다룬 것

입문에서 일부러 미룬, 실무에서 만나는 주제들이에요. 각 카드를 눌러 무엇이고 언제 필요한지 보세요. 지금 다 알 필요는 없어요 — "이런 게 있구나" 정도면 충분하고, 필요할 때 찾아 들어가면 돼요.

→ 이 트랙의 멘탈 모델(요청 여정·async·DI·풀)이 있으면, 이 주제들도 "어디에 붙는 이야기"인지 빨리 자리잡혀요.

Step 03 — Closing

마무리 — 요청 하나의 여정을 읽는 눈

처음엔 "요청을 보내면 응답이 온다" 정도였다면, 이제 그 사이 서버 안에서 무엇이 어떻게 도는지가 보여요 — 어디로 가고(라우팅) · 어떻게 걸러지고(검증·응답모델) · 한 일꾼이 어떻게 수천 요청을 받고(async·이벤트 루프) · 준비물을 어떻게 받고(의존성 주입) · 연결을 어떻게 빌리고(풀) · 누구인지 어떻게 확인하고(인증) · 어떻게 감싸이고(미들웨어) · 어떻게 띄워지는지(워커)까지. FastAPI는 결국 보이지 않던 런타임을 한 줄씩 읽어내는 일이었어요. 그 눈이 생겼다면, 어떤 백엔드 코드 앞에서도 길을 잃지 않을 거예요.
