PART B · 입력과 출력 — CH 03

Pydantic 검증

바깥에서 온 입력은 믿으면 안 돼요 — 빠진 필드, 틀린 타입, 말도 안 되는 값이 수두룩하죠. FastAPI는 Pydantic으로 이걸 문 앞에서 거릅니다. 핸들러가 "이 요청은 이런 모양(name은 문자열, age는 정수…)"이라고 모델로 선언하면, 들어온 JSON을 그 틀에 맞춰보고 — 맞으면 타입이 보장된 깨끗한 객체로 바꿔 핸들러에 넘기고, 안 맞으면 핸들러에 닿기도 전에 무엇이 왜 틀렸는지 적은 422로 돌려보내요. 덕분에 핸들러 안에서는 "혹시 age가 문자열이면?" 같은 방어 코드를 안 짜도 돼요. 검증은 핸들러의 문지기예요.

Step 00 — The Gatekeeper

JSON이 모델 틀을 통과하면

▶를 누르면 들어온 JSON {name, age, email}이 UserModel 문지기 앞에 서요. 문지기는 필드를 하나씩 검사해요 — name은 문자열? age는 정수? email은 형식 맞음? 다 통과하면 타입 있는 객체가 되어 핸들러로. 핸들러는 깨끗한 입력만 받죠.

→ 문자열 "25"도 age:int면 Pydantic이 25(정수)로 변환해줘요(가능하면). 변환 불가하면 막고요.

Step 01 — Break the Input

입력을 직접 망가뜨려 보기

각 필드를 망가뜨려 보세요(타입 틀리기·빼기). 문지기가 어느 필드가 왜 틀렸는지 짚어내요. 하나라도 틀리면 결과는 422 — 그리고 핸들러는 실행조차 안 돼요. 에러 메시지에 필드 위치가 다 들어 있어서, "분명 보냈는데 422"의 원인을 바로 찾을 수 있어요.

→ 422 = "형식은 JSON인데 내용이 모델과 안 맞음". 400(아예 깨진 요청)과는 달라요.

Step 02 — Blocked Before the Handler

핸들러 전에 막는다 — 그래서 핸들러가 깨끗하다

검증이 파이프라인의 어디에 있는지가 핵심이에요. 라우팅 다음, 핸들러 바로 앞이죠. 토글로 입력을 유효/무효로 바꿔보세요 — 무효면 검증 칸에서 막혀 핸들러로 못 가고 422가 바로 나가요. 유효해야만 핸들러가 돌아요. 덕분에 핸들러 안에는 방어 코드가 필요 없어요.

→ "핸들러는 항상 올바른 입력만 받는다"는 보장 — 이게 Pydantic 검증이 주는 가장 큰 선물이에요.

Step 03 — Takeaways

정리 — 검증을 한 문장으로

Pydantic 검증은 들어온 JSON을 모델 틀에 맞춰, 통과하면 타입 보장 객체로·실패하면 핸들러 전에 422로 돌려보내는 문지기예요. 핸들러는 깨끗한 입력만 받죠. 입력을 거르는 법을 봤으니, 이번엔 반대 방향 — 내보내는 것을 거르는 법이에요. 비밀번호 같은 게 실수로 새 나가지 않게요.
