PART C · async와 이벤트 루프 — CH 07 · 핵심

루프를 막지 마라

이벤트 루프의 우아함엔 치명적 약점이 하나 있어요 — 일이 await로 양보할 때만 다음으로 넘어가요. 그런데 어떤 일이 await 없이 CPU를 오래 붙잡으면(무거운 계산, 또는 동기 DB 드라이버 같은 블로킹 호출), 루프는 양보 신호를 못 받아 그 자리에 얼어붙어요. 그러면 그동안 다른 모든 요청이 멈춰요 — 그 요청들은 잘못이 없는데도요. 이게 FastAPI 1순위 실수예요. 함정의 핵심은 async def 안에서 동기 블로킹 호출을 하는 거예요(예: 비동기 드라이버 대신 동기 드라이버, time.sleep, 무거운 반복문). 해법은 둘 — 진짜 비동기 라이브러리를 await하거나, 동기 작업이라면 핸들러를 그냥 def로 두는 거예요(그러면 FastAPI가 알아서 별도 스레드풀로 빼서 루프를 안 막아요). 그래서 — "async = 빠름"이 아니에요. 잘못 쓴 async가 오히려 가장 느려요.

Step 00 — The Loop Freezes

한 요청의 블로킹이 모두를 멈춘다

요청 3개가 들어왔는데, R1이 async def 안에서 동기 블로킹 호출(양보 없는 무거운 일)을 해요. ▶로 재생하면 — R1이 CPU를 붙잡은 빨간 구간 동안 루프가 얼어붙어, 죄 없는 R2·R3가 시작도 못 하고 기다려요. 블로킹이 끝나야 비로소 다음으로 넘어가죠.

→ 빨간 막대 = 양보 없이 CPU를 붙잡은 블로킹. 이 동안 이벤트 루프는 아무 일도 못 해요.

Step 01 — Three Ways

같은 느린 작업, 세 가지 방식

R1이 1초짜리 작업을 하는데, 방식을 바꿔 비교하세요. ① async+await(진짜 비동기 I/O): 기다리며 양보 → R2·R3 멀쩡. ② async+블로킹(동기 호출을 async 안에서): 루프 얼어붙음 → 전부 지연(최악!). ③ def 핸들러: FastAPI가 별도 스레드풀로 빼줘서 루프는 자유 → R2·R3 멀쩡. ①과 ③은 좋고, ②만 나빠요.

→ 핵심 반전: 잘 모를 땐 def가 더 안전해요(FastAPI가 스레드풀로 빼주니까). 어설픈 async def+블로킹이 가장 위험.

Step 02 — The One Rule

규칙 한 문장 — 그리고 "async ≠ 빠름"

각 상황을 눌러 무엇을 써야 하는지 보세요. 외울 한 문장: "async def면 안에서 await만 — 동기 블로킹 작업이면 그냥 def로 둬라(FastAPI가 스레드풀로 빼준다)." 그리고 async는 빠름의 동의어가 아니에요 — 기다림을 메우는 도구일 뿐, 잘못 쓰면 가장 느려요. 무거운 CPU 계산은 async로도 안 빨라지고(병렬 아님), 오히려 루프를 막죠.

→ 참고: 스레드풀도 수가 한정돼 있어요(기본 수십 개). def 핸들러가 폭주하면 그 한계에 부딪혀요 — 만능은 아니에요.

Step 03 — Takeaways

정리 — 루프를 막지 마라

async def 안의 동기 블로킹은 이벤트 루프를 얼려 모든 요청을 멈춰요 — FastAPI 1순위 실수. 해법: 진짜 비동기 라이브러리를 await하거나, 동기 작업이면 핸들러를 def로(스레드풀행). "async = 빠름"은 미신이에요. 자, 그럼 제대로 쓴 비동기 서버는 한 루프로 얼마나 많은 요청을 받아낼까요? 그리고 "응답 후에 도는" 백그라운드 작업의 정체는? 다음 챕터예요.
